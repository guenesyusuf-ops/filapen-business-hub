Die vier Wetten, auf denen der Plan basiert

  • 1. Filapen zuerst, SaaS danach. Wir bauen es als Single-Tenant-MVP für Filapen, refactorieren ab Phase 4 zu Multi-Tenant. Das halbiert das MVP-Risiko und gibt uns bis dahin 3 Monate echte Produktionsdaten als Case-Study.
  • 2. Meta-Attribution ist die Killer-Funktion. Die Daten zeigen: Meta macht bei dir aktuell −€128 Wochenprofit, Google +€6.745. Wenn wir Meta über sauberen CAPI-Dedup + Server-Side Click-Reconciliation +10–15 % effizienter kriegen, ist das die Story, mit der wir auch andere DACH-Advertiser überzeugen.
  • 3. EU-Hosting als Differenzierungs-Vorteil. Hyros ist US-hosted. Für DACH-Kunden mit DSGVO-Angst ist eine Frankfurt-hosted First-Party-Alternative ein harter USP — und du als deutsche Domain filapen.de kannst das als Referenz nach vorn stellen.
  • 4. Claude-Vibe-Coding als Execution-Hebel. Ein 1-Personen-Team (du + Claude) kann das in 12 Wochen bauen, wenn der Scope diszipliniert bleibt. Wir bauen nur was du in den 40 Screenshots tatsächlich nutzt — nicht alles, was Hyros kann.

Technische Foundation-Entscheidungen im Detail

Warum Cloudflare Workers und nicht Vercel Edge Functions?

Vercel Edge Functions laufen auf der gleichen Cloudflare-V8-Runtime, aber mit Vercel-Overhead. Für reines Ingestion willst du den direkten Weg. Außerdem: CF erlaubt Custom-Domains ohne Vercel-Eigentümerschaft, was für die Custom-Tracking-Domains der Kunden (t.kunde.de) entscheidend ist — jede CNAME landet direkt auf unserem Worker ohne Vercel-Proxy-Schicht.

Warum Supabase statt „richtigem" Postgres auf Hetzner?

Für ein 1-Personen-Team ist Supabase Managed Auth + Row-Level-Security + Realtime + Storage ein 6-Monats-Vorsprung. Wenn wir in Phase 4 Performance-Probleme bekommen, migrieren wir den OLTP auf CrunchyBridge oder Neon EU — aber bis dahin nicht. Trade-off ist akzeptabel.

Warum ClickHouse und nicht TimescaleDB / DuckDB / Parquet-on-S3?

Hyros macht Source-Stats-Aggregate über Millionen Events in Millisekunden. Das ist exakt ClickHouse-Terrain. Timescale ist für Time-Series gut, aber bei Multi-Dimension-Slicing (User × Channel × Campaign × Day) wird es teuer. DuckDB/Parquet wäre billiger, aber für Live-Dashboards zu langsam. ClickHouse Cloud EU gibt es ab ~$70/Monat — vertretbar.

Warum nicht einfach Fivetran + dbt + Metabase?

Weil das eine Analytics-Lösung ist, kein Tracker. Wir sind die Datenquelle, nicht Konsument. Ein BI-Stack löst dir „zeige mir Zahlen", aber nicht „nimm den fbc-Cookie, match ihn über Identity Graph, push an Meta CAPI mit Dedup". Das ist Produktcode, nicht Analytics-Pipeline.

Warum Hono statt reinem Express / Fastify?

Hono läuft als einziger Framework-Code sowohl in Cloudflare Workers als auch in Node — das heißt: gleicher Validator, gleiche Middleware, gleiche Types für Edge-Ingest-Endpoint und Backend-API. Weniger Code, weniger Divergenz-Bugs.

Die 5 Migrationsschritte

Schritt 1 · Script-Doppel-Install (Tag 1)

Beide Tracker laufen gleichzeitig auf filapen.de. Hyros bleibt auf t.filapen.de. Wir installieren zusätzlich unser Script auf einer zweiten Sub-Domain t2.filapen.de. Jeder Event wird zweimal gesendet — einmal an Hyros, einmal an uns.

<!-- Filapen-Script (dein aktuelles) -->
<script>
var head = document.head;
var s = document.createElement('script');
s.src = "https://t.filapen.de/v1/lst/universal-script?ph=e6793dc…&tag=!clicked&ref_url=" + encodeURI(document.URL);
head.appendChild(s);
</script>

<!-- NEU: Filapen Tracking Platform parallel -->
<script>
var s2 = document.createElement('script');
s2.src = "https://t2.filapen.de/v1/script?ph=<neuer_hash>&tag=!clicked&ref_url=" + encodeURI(document.URL);
head.appendChild(s2);
</script>

Zero Risiko. Wenn unser Script crasht, läuft Hyros weiter. Wenn Hyros ein Problem hat, beeinflusst es uns nicht.

Schritt 2 · Historische Daten aus Hyros ziehen (Woche 2 – 3)

Hyros hat einen Full Data Export (CSV / JSON) für Sales, Refunds, Leads, Calls, UTM-Touches. Wir:

  • Laden den Export als Rohdaten in Hetzner S3
  • Schreiben einen TypeScript-Importer, der Records auf unser Schema mappt (Identity, Event, Sale, Refund)
  • Identity-Matching: E-Mail-Hash zuerst, dann fbp/gclid wenn vorhanden, dann IP+UA-Fingerprint als Fallback
  • Fehlende Attribution-Chains werden aus UTM-Feldern rekonstruiert (Last-Click Minimum-Garantie)
  • Abschlusstest: Aggregate Revenue pro Kanal gegen Hyros-Zahlen — muss ± 0,5 % stimmen

Schritt 4 · CNAME-Switch (Woche 8, nach Dashboard-Abnahme)

An einem Dienstag 8 Uhr (niedrigstes Traffic-Fenster der Woche):

  • 1. DNS-TTL auf t.filapen.de vorab auf 60 Sek reduzieren (3 Tage vorher)
  • 2. CNAME-Target umstellen: von Hyros-Endpoint auf unseren Worker
  • 3. Monitoring: Event-Durchsatz sollte 100 % auf unseren Endpoint fallen innerhalb 5 Min
  • 4. Rollback-Plan: CNAME zurücksetzen, TTL reicht für 60-Sek-Rollback
  • 5. Hyros-Script-Tag bleibt noch 7 Tage im HTML — als Safety-Net, falls doch etwas schief geht
Go/No-Go-Checklist vor Switch: Dashboard-Drift 7 Tage lang < 3 %, keine P1-Bugs offen, Alerting aktiv, Rollback-Befehl dokumentiert, Shopify-Webhooks bestätigt.

Schritt 5 · Hyros-Kündigung (Woche 10)

Nach 2 Wochen stabilem Cut-Over:

  • Hyros-Script-Tag aus allen Filapen-Templates entfernt
  • Hyros-Billing im April/Mai gekündigt — schriftlich, mit Bezug auf die letzte Abrechnung $3.539
  • Vollständiger Hyros-Datenexport als Archiv in S3 (Compliance-Retention 7 Jahre)
  • Post-Mortem-Review: was lief gut, was müssen wir für SaaS-Onboarding neuer Kunden anders machen?

Wie Claude-Vibe-Coding konkret aussieht

Prinzip: Jeder Sprint ist eine einzelne Feature-Flag-baren Pull-Request. Claude bekommt als Input den Sprint-Scope + Links zu relevanten Files + die bestehende Code-Basis. Output ist Code + Migrations + Tests + kurzer Deploy-Bericht. Wöchentlich eine Demo gegen echte Filapen-Daten.

Was du (Filapen) im Wochen-Zyklus tust

  • Montag: Sprint-Scope mit Claude durchsprechen + Blocker klären
  • Mo – Fr: Claude arbeitet, du reviewst PRs & klärst Product-Fragen
  • Freitag: Demo gegen echte Daten, was funktioniert/nicht
  • Sonntag: Rückblick + nächster Sprint-Scope festlegen

Wo ein zweiter Mensch helfen sollte (später)

  • Ab Phase 3: Ein Part-Time-DevOps-Mensch für ClickHouse-Ops + Alerting
  • Ab Phase 4: Ein Support-/CS-Mensch für die ersten externen Kunden
